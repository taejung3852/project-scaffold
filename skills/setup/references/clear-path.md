# clear-path — CLEAR 경로 상세 절차

`ambiguity-check.md`에서 **CLEAR** 또는 **FENCE**로 분류된 항목의 처리 절차.

---

## Step 1 — 두 필터 (질문 생략 여부 결정)

순서대로 적용한다. **둘 다 통과한 것만 실제 질문으로 제시한다.**

```
필터 1. Step 0.5 스캔 결과로 답할 수 있나?
  → YES: 질문 생략. "(감지: pytest) 이걸로 확정할게요." 제시 후 확인만

필터 2. 합리적인 기본값(defensible default)으로 답할 수 있나?
  → YES: 기본값 채택. "기본값으로 [X]로 설정했어. 다르면 말해줘." 기록 후 진행

→ 둘 다 NO: 진짜 질문으로 제시 (WHY 한 줄 함께)
```

---

## Step 2 — 평가 기준

답변이 다음 조건을 **모두** 충족해야 통과다:

| 조건 | 판정 기준 |
|---|---|
| **구체성** | 두 개발자가 동일하게 해석할 수 있는가? |
| **실행 가능성** | 에이전트가 추가 질문 없이 따를 수 있는가? |
| **예시 가능성** | 올바른 예와 잘못된 예를 들 수 있는가? |
| **완결성** | 엣지케이스가 명확한가? |

---

## 모호한 답변 패턴 (즉시 follow-up)

- **과도한 유연성**: "상황에 따라", "적절히", "필요시", "가능하면"
- **재진술**: 질문을 그대로 돌려준 답변 ("네이밍은 읽기 쉽게")
- **미완성**: "아직 모르겠어", "나중에 정할게"
- **너무 짧음**: 단어 1~2개, 맥락 없는 답변
- **과도한 일반화**: "일반적인 방식으로", "표준적으로"

---

## Follow-up 방식

| 상황 | 전략 |
|---|---|
| 모호한 표현 | "'{표현}'이 구체적으로 어떤 의미야? 예시 하나만 들어줄 수 있어?" |
| 재진술 | "그 규칙을 따를 때 어떻게 보여야 해? 올바른 예 하나만 들어줘" |
| 미완성 | "지금 결정 안 해도 되는데, 가장 흔한 케이스 기준으로 일단 정해볼까?" |
| 너무 짧음 | "조금 더 풀어서 말해줄 수 있어? 이유도 같이 알고 싶어" |

같은 질문 반복 금지. 답변 내용을 기반으로 구체화를 유도한다.

**루프 가드**: follow-up 2회 초과 후에도 모호하면 → `references/unclear-path.md`의 TBD 처리로 위임.

## 예시

**네이밍 컨벤션:**
- ❌ "읽기 쉽게 써줘" → "읽기 쉽다는 게 어떤 형식이야? snake_case? camelCase? 예시 하나만"
- ✅ "Python이니까 변수·함수는 snake_case, 클래스는 PascalCase, 상수는 UPPER_SNAKE_CASE"

**Git 컨벤션:**
- ❌ "커밋 잘 써줘" → "커밋 메시지 형식이 어때야 해? feat:/fix: 같은 접두사가 있어?"
- ✅ "feat:/fix:/chore: + 콜론 + 공백 + 한국어 설명. 예: feat: 로그인 기능 추가"

**TDD:**
- ❌ "테스트는 꼼꼼하게" → "꼼꼼하다는 게 커버리지 몇 %야? 단위·통합·E2E 중 어디까지?"
- ✅ "pytest, 커버리지 80% 이상, 단위+통합까지. E2E는 나중에"
